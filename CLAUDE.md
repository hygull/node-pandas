## Project

`node-pandas` is a published npm package that brings pandas-like data manipulation (Series, DataFrame, GroupBy, CSV I/O, window operations, etc.) to Node.js. The library has zero runtime dependencies ‚Äî only `jest` as a devDependency. Public entry is `src/index.js`.

## Commands

```bash
npm test                  # run full Jest suite
npm run test:watch        # watch mode
npm run test:coverage     # generates coverage/ (80% threshold enforced for statements/branches/functions/lines)
npm run test:unit         # tests/unit only
npm run test:integration  # tests/integration only
npm run test:property     # tests/property only

# Run a single test file
npx jest tests/unit/series.test.js

# Run tests by name pattern
npx jest -t "sort_values"
```

There is no build step (plain CommonJS, `main: ./src/index.js`) and no lint script configured.

## Architecture

The library is organized around two `Array`-extending classes plus a `GroupBy` helper. **Both `Series` and `DataFrame` extend `Array`** ‚Äî this is load-bearing: it gives `length`, indexing (`s[0]`), and iteration for free, but means anything returning a new instance must construct via `new Series(...)` / `new DataFrame(...)` rather than relying on inherited `Array.prototype` methods (which would return plain Arrays).

### Key cross-file relationships

- `dataframe.js` imports `Series` and constructs Series instances when columns are accessed (e.g. `df.name` returns a Series, not a raw array).
- `dataframe.js` instantiates `GroupBy` from `features/GroupBy.js`; `GroupBy` in turn imports `Series` to build aggregated columns. There is a circular-import risk here ‚Äî `readCsv` is intentionally defined inline in `src/index.js` rather than in `utils/` to avoid one (see comment at top of `index.js`).
- All new methods on Series/DataFrame should propagate `_index` and `_name` to the returned Series (see the rolling/expanding/cum* methods for the pattern).
- Null handling is centralized: use `isNull(val)` from `utils/typeDetection.js` rather than ad-hoc `== null` checks. Numeric methods should `toNumeric(val)` and skip nulls, preserving them in output positions.
- Errors should be thrown as the typed classes from `utils/errors.js` with a `context` object (`operation`, `value`, `column`, `expected`, `actual`) ‚Äî these power formatted `toString()` output and suggestions.

### Test layout

Jest config (`jest.config.js`) only matches `**/tests/**/*.test.js`. Coverage collection is **scoped to `src/series` and `src/dataframe` only** ‚Äî `src/utils`, `src/features`, `src/bases`, `src/messages`, and `src/index.js` are excluded from coverage thresholds. Reusable helpers live in `tests/utils/testDataGenerator.js` and `tests/utils/testAssertions.js`; prefer these over hand-rolled fixtures.

The existing tests under `tests/unit/` and the helpers in `tests/utils/` were generated by Kiro from the specs under `.kiro/specs/`. When adding or changing features, follow the same patterns and keep tests in sync with the spec changes (update `tasks.md`/`requirements.md` first, then mirror in tests).

## Conventions specific to this repo

- **Commit format is Jira SMART**: `TICKET-ID #time Xm #comment <message>`, optionally prefixed with a conventional type (`feat(series): DSA-4 #time 30m #comment ...`). See `.kiro/agents/git-committer-agent.md` and recent `git log` for examples. The `/cap` and `/smart-commit` skills produce this format.

- **Specs live under `.kiro/specs/`** (`pandas-like-enhancements/`, `comprehensive-pandas-features/`) ‚Äî check `tasks.md` and `requirements.md` there before adding large features so work stays aligned with the planned roadmap.
- **JSDoc is expected** on every public method on `Series`/`DataFrame`/`GroupBy`, including `@example` blocks. The existing methods set the bar ‚Äî match that depth for new ones.
- **Versioning follows semver**, tracked in `CHANGELOG.md`. Current version is 2.2.0.
